# Pin Plate Product Plan

이 문서는 Pin Plate의 제품 방향, 차별화 전략, 기능 우선순위를 정리한 공식 기준 문서다.

기존 `.claude/plans/` 하위 계획 문서는 `.gitignore` 대상이므로, 앞으로 개발자가 항상 참고해야 하는 제품 기획 기준은 이 루트 문서로 관리한다.

## 1. 제품 개요

### 서비스 정의

Pin Plate는 내가 다녀온 장소와 가보고 싶은 장소를 지도 위에 핀으로 저장하고, 별점과 메모, 이미지를 함께 남기는 개인 장소 아카이빙 서비스다.

### 제품 의도: 비로그인 작성과 업로드

Pin Plate의 핵심 제품 의도는 사용자가 가입/로그인 장벽 없이 장소 기록을 바로 시작할 수 있게 하는 것이다. 따라서 **비로그인 사용자도 글 작성과 사진 업로드가 가능해야 한다.**

이 원칙은 보안 보완이나 인증 정책을 설계할 때도 유지한다. 비로그인 업로드 보안을 해결한다는 이유로 `/api/image` 같은 업로드 진입점을 단순히 로그인 전용으로 닫아서는 안 된다. 대신 게스트 세션, 브라우저/기기 기준 임시 식별자, 업로드 개수/용량 제한, rate limit, 만료 시간, 파일 타입 검증, 악성 파일 검증, 사후 정리 정책처럼 비로그인 사용을 유지하면서 남용을 줄이는 방식으로 설계한다.

비로그인 데이터 정책은 다음과 같다.

- 비로그인 사용자는 장소 글과 사진을 임시로 작성/업로드할 수 있다.
- 비로그인 작성 데이터는 기본적으로 현재 브라우저/기기 기준의 임시 데이터로 본다.
- 웹은 서버 발급 HttpOnly 게스트 업로드 세션 쿠키를 사용하고, 모바일 네이티브 업로드는 secure storage에 저장한 서버 발급 게스트 세션 토큰을 요청 헤더로 보내는 방식을 사용한다.
- 게스트 업로드 세션은 클라이언트가 임의로 만들 수 없는 서버 서명 토큰이어야 하며, 웹과 모바일은 같은 서버 검증과 제한 정책을 공유한다.
- 영구 저장, 기기 간 동기화, 계정 귀속, 장기 보존은 가입/로그인 이후에 제공한다.
- 가입 유도는 작성과 업로드를 막는 장벽이 아니라, 이미 만든 기록을 보존하기 위한 제안이어야 한다.
- 보안 리포트나 QA에서 비로그인 업로드 위험이 발견되면, 제품 의도를 깨는 인증 차단보다 게스트 업로드 보호 장치를 우선 검토한다.

### 포지셔닝

Pin Plate는 공개 리뷰 서비스보다 개인 취향 기반 장소 지도에 가깝다. 사용자는 현재 위치 주변의 맛집과 카페를 발견하고, 마음에 드는 곳을 내 지도에 저장하며, 쌓인 기록을 기준으로 다시 찾고 공유한다.

제품 방향은 다음 문장으로 정리한다.

> 주변 맛집과 카페를 발견하고, 내 취향 지도에 저장해 다시 활용하는 지도 서비스

### 핵심 가치

- 지도 위에서 내 장소 기록을 한눈에 본다.
- 카카오 장소 검색으로 정확한 장소를 선택한다.
- 현재 위치 주변의 맛집과 카페를 발견한다.
- 별점, 메모, 이미지로 개인 경험을 기록한다.
- 위시리스트와 방문 기록을 구분해 다음 행동을 정한다.
- 태그, 상태, 위치를 기준으로 저장 장소와 주변 후보를 비교한다.
- 웹과 모바일 앱에서 모두 사용할 수 있고, 앱에서는 네이티브 위치 정보를 활용한다.

## 2. 차별화 전략

### Google Maps 대비

Google Maps는 장소 검색, 길찾기, 공개 리뷰, 영업 정보 확인에 강하다. Pin Plate가 같은 영역에서 경쟁하기보다 집중해야 할 방향은 다음과 같다.

- 현재 위치 주변의 음식점/카페 후보를 빠르게 발견하기
- 발견한 장소를 내 지도에 저장하고 상태와 태그로 관리하기
- 공개 평점보다 내 저장 여부, 내 경험, 내 취향 맥락을 함께 보여주기
- "지금 근처에서 어디 가지?", "여기 저장해 둘까?", "여기 어땠지?"에 답하기
- 지인에게 내 장소 큐레이션을 공유하기

### Mapstr 대비

Mapstr는 개인 지도 저장과 태그 기반 분류에 강점이 있다. Pin Plate도 개인 지도 아카이빙이라는 관점에서는 유사하지만, 국내 사용자를 기준으로 장소 검색 정확도, 모바일 위치 경험, 맛집/카페 기록 맥락에 더 집중할 수 있다.

Pin Plate는 범용 장소 저장보다 음식점, 카페, 방문 경험 중심의 장소 취향 기록으로 좁히는 편이 좋다.

### 장소별 방문 히스토리

Pin Plate는 한 장소를 한 번 저장하고 끝내는 장소 북마크 앱이 아니라, 같은 장소에 여러 번의 방문 기록을 쌓을 수 있는 개인 히스토리 앱에 가깝다. 같은 음식점이나 카페라도 방문 시점, 동행, 메뉴, 요일, 날씨, 혼잡도에 따라 경험이 달라질 수 있다.

따라서 한 장소에 여러 게시글을 허용하는 것은 중복 리뷰가 아니라 제품 차별화 요소로 다룬다. 장소는 지도와 추천의 기준이 되고, 게시글은 그 장소에 쌓인 방문 경험의 타임라인이 된다. 사용자는 "여기 어땠지?"를 단일 평점이 아니라 내 과거 방문 기록의 흐름으로 확인할 수 있어야 한다.

### 위치 기반 탐색 서비스 대비

neer처럼 위치 기반으로 주변 장소를 탐색하는 서비스와 비교하면, Pin Plate도 현재 위치 주변의 맛집과 카페 발견을 반복 사용의 시작점으로 삼아야 한다. 다만 차별점은 공개 장소를 보여주는 데서 끝나는 것이 아니라, 발견한 장소를 `가보고싶음`, `다녀옴`, `다시가고싶음` 같은 개인 상태와 태그로 저장해 이후 의사결정에 다시 쓰게 만드는 데 있다.

핵심은 "내가 이미 저장한 곳만 다시 추천"하는 것이 아니다. 초기 사용자에게 저장 장소 기반 추천은 빈 화면이나 단순 필터에 가까울 수 있다. V1의 추천은 주변 공개 장소 후보를 기본으로 보여주고, 내 저장 데이터는 이미 저장한 곳 표시, 취향 태그 반영, 재방문 후보 강조 같은 보정 데이터로 사용한다.

## 3. 사용자 문제와 반복 사용 흐름

### 고객 입장에서 자주 쓸 이유

사용자가 Pin Plate를 반복해서 열 이유는 기록만으로는 부족하다. 기록이 다시 쓰이는 순간을 만들어야 한다.

- 약속 장소를 정할 때, 내가 저장한 곳 중 괜찮았던 곳을 빠르게 찾는다.
- 현재 위치 근처에서 갈 만한 맛집이나 카페를 발견한다.
- 마음에 드는 주변 장소를 바로 `가보고싶음`으로 저장한다.
- 현재 위치 근처에서 예전에 저장해 둔 맛집이나 카페도 함께 확인한다.
- 친구에게 특정 동네의 추천 장소만 모아 공유한다.
- 최근 방문 기록을 보며 내가 어떤 음식을 자주 먹는지, 어떤 지역을 자주 가는지 돌아본다.
- 가보고 싶은 곳과 이미 다녀온 곳을 구분해 다음 방문 후보를 정한다.
- 별점과 메모를 통해 다시 갈 곳과 한 번이면 충분한 곳을 구분한다.

반복 사용과 외부 유입의 핵심 성장 루프는 다음과 같다.

1. 친구나 지인이 공유한 맛집 지도를 연다.
2. 마음에 드는 장소를 가입 없이 현재 브라우저/기기의 내 지도에 저장한다.
3. 저장한 장소를 상태와 태그로 분류하고, 실제 방문 후 기록을 남긴다.
4. 저장한 장소를 다시 찾거나 주변 추천에서 비교한다.
5. 내가 고른 장소 묶음을 다시 공유한다.

현재는 직접 저장과 일부 다시 찾기에 가까우므로, V1은 `공유 지도에서 내 지도가 시작되는 순간`을 반복 사용과 확산의 첫 루프로 검증한다.

### 고객 입장에서 아쉬운 점

#### 장소가 쌓일수록 다시 찾기 어려워질 수 있음

핀과 리스트만으로는 저장 장소가 많아졌을 때 탐색 비용이 커진다. 사용자는 "성수에서 조용한 카페", "비 오는 날 갈 만한 곳", "친구와 저녁 먹기 좋은 곳"처럼 상황 단위로 찾고 싶어질 가능성이 높다.

#### 위시리스트와 방문 기록의 의미가 다름

가보고 싶은 곳과 실제 다녀온 곳은 사용 목적이 다르다.

- 위시리스트: 다음 행동을 유도하는 후보군
- 방문 기록: 회고, 추천, 재방문 판단의 근거

두 데이터가 같은 흐름에 섞이면 지도는 단순해지지만, 사용자의 의사결정은 흐려질 수 있다.

#### 내 취향이 쌓이고 있다는 느낌이 약할 수 있음

별점과 메모를 저장해도, 서비스가 이를 해석해 주지 않으면 사용자는 내가 데이터를 쌓는 이유를 체감하기 어렵다. 기록이 리포트, 추천, 필터, 공유 지도 등으로 되돌아와야 한다.

#### 공유 가치가 제한적일 수 있음

장소 기록은 개인용으로도 유용하지만, 맛집과 카페 기록은 신뢰하는 지인에게 공유될 때 가치가 커진다. 특정 동네, 카테고리, 태그만 골라 공유할 수 있으면 받은 사람은 검색 없이 바로 후보를 고르고, 마음에 드는 장소를 자기 지도에 저장할 수 있다.

## 4. 기능 후보

### 태그/상황별 분류

장소에 사용자가 직접 태그를 붙일 수 있게 한다. 태그가 생기면 지도 필터, 리스트 필터, 공유 지도, 추천 기능, 취향 리포트의 기반이 된다.

초기에는 자유 태그보다 추천 태그 선택을 우선한다. 태그의 중심은 음식 종류보다 목적, 상황, 판단에 둔다. Pin Plate가 단순 맛집 카테고리 저장 앱이 아니라 내 취향과 맥락을 다시 꺼내 쓰는 서비스가 되기 위해서다.

### 장소별 방문 기록

같은 장소에 여러 게시글을 작성할 수 있게 한다. 이때 게시글은 별도의 장소 중복 데이터가 아니라, 하나의 장소에 연결된 방문 기록으로 이해되어야 한다.

지도와 리스트에서는 장소를 기본 단위로 보여주고, 장소 상세에서는 해당 장소의 방문 기록을 시간순으로 보여준다. 여러 번 방문한 장소는 최근 기록, 대표 태그, 방문 횟수, 평균 또는 최근 별점 같은 요약 정보를 통해 다시 찾기 쉽게 만든다.

V1에서는 복잡한 통계보다 "이 장소에 N번째 기록"이라는 맥락을 명확히 보여주는 것을 우선한다.

### 위시리스트와 방문 기록 분리

장소 상태를 명확히 나눈다.

- 가보고싶음
- 다녀옴
- 다시가고싶음
- 추천하고싶음

별점 0점을 위시리스트처럼 쓰는 방식은 내부적으로 간단하지만, 제품 경험에서는 의미가 덜 명확하다. 사용자가 "여기는 아직 안 가봤지만 저장해 둔 곳"이라는 상태를 직접 이해할 수 있어야 한다.

### 현재 위치 기반 주변 맛집 탐색

현재 위치를 기준으로 주변의 음식점, 카페, 베이커리 같은 공개 장소 후보를 보여준다. 사용자는 추천 결과에서 마음에 드는 곳을 바로 `가보고싶음`으로 저장하거나, 이미 저장한 장소라면 내 상태와 기록을 확인한다.

내 저장 장소 기반 추천은 V1 핵심이 아니라 보조 기능으로 둔다. 저장 장소가 충분히 쌓인 사용자에게는 근처에 있는 `다시가고싶음`, 높은 별점 장소, 특정 태그 장소를 함께 강조한다.

### 공유 가능한 맛집 지도

내 장소 전체가 아니라 일부만 골라 공유할 수 있게 한다. 특정 태그, 지역, 상태, 선택 장소 기준으로 공유 지도를 만들고, 공개 링크로 접근할 수 있게 한다.

공유는 불특정 다수에게 리뷰를 노출하는 소셜 피드가 아니라, 신뢰하는 지인의 장소 묶음을 내 지도에 가져오는 흐름이다.

### 내 취향 리포트

사용자의 기록을 요약해 보여준다. 단순 통계가 아니라 다음 행동과 연결되어야 한다.

- "이번 달에는 성수 카페를 많이 저장했어요"
- "5점 준 일식집이 많아요"
- "다시 갈 만한 곳 7개가 근처에 있어요"

## 5. V1 기능 우선순위

### 평가 기준

각 기능은 5점 만점으로 평가한다. 점수는 현 문서의 제품 논리를 기준으로 한 문서 수준의 판단이며, 실제 개발 공수 추정은 별도 코드 분석과 구현 계획에서 다시 산정한다.

- 사용자 반복 사용 기여도: 사용자가 앱을 다시 열 이유를 만드는 정도
- 다른 기능의 기반 데이터 여부: 필터, 추천, 공유, 리포트의 선행 데이터가 되는 정도
- 현재 제품 방향과의 차별화 기여도: 공개 리뷰/일반 지도와 다른 가치를 만드는 정도
- 외부 유입/확산 기여도: 내 사용자가 아닌 사람이 Pin Plate를 처음 만나는 계기를 만드는 정도
- 구현 난이도 대비 효과: V1에서 비교적 빠르게 체감 가치를 만들 수 있는 정도
- V1에서 검증 가능한 정도: 출시 후 사용 여부와 효과를 관찰하기 쉬운 정도

## 6. 기능별 상세 정책

### 태그 전략

태그는 장소를 다시 찾고, 필터링하고, 추천하고, 공유하기 위한 핵심 데이터다. Pin Plate의 기본 태그 단위는 게시글이 아니라 장소다. 사용자가 같은 장소에 여러 게시글을 작성하더라도, 지도 필터, 리스트 필터, 추천, 공유 지도는 장소 태그를 기준으로 동작한다.

MVP에서는 추천 태그 목록을 프론트 상수로 관리하고, 사용자가 선택한 태그 id/slug를 안정적으로 저장한다. 게시글마다 태그를 처음부터 다시 입력하게 만들지 않고, 장소에 이미 저장된 태그를 재사용하는 흐름을 기본으로 둔다.

음식 종류는 기본 분류로 유지하되, 제품 차별화의 중심은 목적, 상황, 판단 태그에 둔다.

#### 장소 태그와 방문 기록 태그

태그는 장소 태그와 방문 기록 태그로 나눈다.

장소 태그는 해당 장소를 다시 찾기 위한 기본 분류다. `카페`, `조용한`, `작업하기좋음`, `데이트`, `가성비`처럼 장소 자체의 성격을 설명하는 태그는 장소에 귀속한다.

방문 기록 태그는 특정 방문에만 해당하는 맥락을 남기기 위한 보조 데이터다. 예를 들어 평소에는 `조용한`, `작업하기좋음`인 카페라도 주말 방문 기록에는 `웨이팅 있음`, `활기찬`, `비 오는 날` 같은 태그가 붙을 수 있다.

V1의 기본 UX는 다음 원칙을 따른다.

- 처음 저장하는 장소에서는 추천 태그 중 장소 태그를 선택한다.
- 같은 장소에 새 게시글을 작성할 때는 기존 장소 태그를 기본값으로 보여준다.
- 사용자는 태그를 매번 처음부터 다시 고르지 않는다.
- 이번 방문에서만 달라진 맥락이 있을 때 방문 기록 태그를 추가하거나 조정한다.
- 장소 태그를 바꾸면 이후 지도 필터, 추천, 공유 기준에도 반영된다.

#### 태그 운영 원칙

- MVP에서는 추천 태그 목록을 프론트 상수로 관리하고, 선택된 태그 id/slug를 저장한다.
- 추천, 지도 필터, 리스트 필터, 공유 지도에서 쓰는 기본 태그는 장소 태그로 둔다.
- 방문 기록 태그는 특정 방문 기록의 맥락을 보조하는 선택 데이터로 둔다.
- 같은 장소에 새 게시글을 작성할 때는 기존 장소 태그를 기본값으로 재사용해 입력 피로도를 줄인다.
- 사용자가 태그를 수정할 때는 장소 태그를 바꾸는 것인지, 이번 방문 기록에만 태그를 남기는 것인지 UI에서 구분되어야 한다.
- 직접 입력 태그는 이후 확장으로 둔다.
- 저장 값은 `혼밥`, `조용한` 같은 label이 아니라 `solo`, `quiet` 같은 안정적인 id/slug를 사용한다.
- 태그 표기는 저장 모델과 UI에서 일관되게 관리한다. 예를 들어 `작업/공부`와 `작업하기좋음`, `사진 잘 나오는`과 `사진맛집`처럼 같은 의미의 태그가 섞이지 않도록 별칭 또는 마이그레이션 기준을 둔다.

### 방문 기록 정책

장소와 게시글의 역할을 분리한다.

- 장소: 지도 핀, 필터, 추천, 공유의 기본 단위
- 게시글: 특정 시점의 방문 경험, 별점, 메모, 이미지, 방문 당시 맥락

한 장소에는 여러 게시글이 연결될 수 있다. 동일한 `kakao_place_id`를 가진 게시글은 같은 장소의 방문 기록으로 묶어 보여준다.

사용자가 이미 저장한 장소를 다시 기록할 때는 "새 장소 추가"가 아니라 "이 장소에 방문 기록 추가" 흐름으로 표현한다. 장소 상세에서는 방문 기록을 최신순 또는 시간순으로 탐색할 수 있어야 한다.

확인해야 할 주요 시나리오:

- 같은 장소에 두 번째 게시글을 작성할 때 기존 장소 태그가 기본으로 표시된다.
- 사용자가 태그를 다시 선택하지 않아도 새 방문 기록을 저장할 수 있다.
- 지도 필터에서 태그를 선택하면 같은 장소가 여러 핀으로 중복 표시되지 않는다.
- 장소 상세에서는 같은 장소의 여러 방문 기록을 시간순으로 볼 수 있다.
- 특정 방문에만 다른 태그를 붙인 경우, 상세 기록에서는 그 차이가 드러난다.

### 상태 정책

UI에서는 위시리스트를 별점이 비어 있는 예외 상태가 아니라, 사용자가 직접 보고 필터링할 수 있는 장소 상태로 다룬다.

- 지도와 리스트 상단에는 같은 상태 필터를 제공한다: `전체`, `가보고싶음`, `다녀옴`, `다시가고싶음`.
- 상태 필터를 바꾸면 지도 핀과 리스트 항목이 함께 바뀐다. 지도와 리스트가 서로 다른 조건으로 보이면 사용자가 현재 보고 있는 장소 집합을 이해하기 어렵다.
- `가보고싶음` 장소는 별점 대신 위시 상태 배지를 표시한다. 방문 전 장소에는 별점 입력을 강제하지 않는다.
- `가보고싶음` 장소를 실제 방문한 뒤 `다녀옴`으로 전환하면서 별점, 메모, 사진을 채우는 흐름을 기본 UX로 둔다.
- `다시가고싶음`은 방문 기록 중 재방문 의사가 있는 장소를 다시 찾기 쉽게 만드는 상태다. 별점과 별개로 사용자가 다음 행동을 결정하는 기준으로 쓴다.
- 기존 회색 핀은 "0점"의 시각 표현이 아니라 "아직 안 가본 곳"의 시각 표현으로 재정의한다.

핀 스타일도 상태 의미를 드러내야 한다. `가보고싶음`은 별점 기반 색상과 분리된 위시 전용 핀 스타일을 사용하고, `다녀옴`과 `다시가고싶음`은 방문 기록 기반의 색상이나 강조 규칙을 사용한다. 이렇게 해야 사용자는 지도만 보아도 아직 가볼 후보와 실제 경험 기록을 구분할 수 있다.

### 추천 정책

V1 추천은 "내가 저장한 장소를 다시 보여주는 기능"이 아니라 "현재 위치 주변의 갈 만한 맛집/카페를 발견하고 내 지도에 저장하는 기능"으로 정의한다. 복잡한 알고리즘보다 외부 장소 API 결과, 거리, 카테고리, 내 저장 여부를 조합해 시작한다.

개인 저장 데이터는 추천의 출발점이 아니라 보정 데이터로 사용한다. 저장 장소가 적은 신규 사용자도 추천 화면에서 주변 후보를 볼 수 있어야 하고, 저장 데이터가 많은 사용자는 이미 저장한 곳과 새 후보를 함께 비교할 수 있어야 한다.

추천 기준:

- 거리: `500m`, `1km`, `3km`, `5km` 중 선택하고, 기본값은 `1km`로 둔다.
- 외부 장소 후보: 현재 위치 주변의 음식점, 카페, 베이커리 등 식음료 중심 장소를 기본 추천 대상으로 둔다.
- 내 저장 여부: 추천 결과가 이미 저장된 장소라면 `가보고싶음`, `다녀옴`, `다시가고싶음` 상태를 표시한다.
- 개인 보정: 내 저장 장소의 태그, 상태, 별점을 활용해 이미 저장한 곳, 다시 갈 곳, 내가 자주 저장한 카테고리를 강조한다.
- 태그/상황: V1 기본 태그 중 `카페`, `술집`, `혼밥`, `데이트`, `작업하기좋음`, `조용한`, `가성비`, `특별한날` 등을 저장 후 분류와 보정 기준으로 사용한다.

주변 탐색 필터:

- 거리: 현재 위치 기준 반경을 선택한다.
- 카테고리: 음식점, 카페, 베이커리 등 식음료 중심 후보를 선택한다.
- 내 저장 여부: 이미 저장된 장소와 아직 저장하지 않은 외부 장소 후보를 구분해 표시한다.

V1 제외 항목:

- 영업 중 여부는 외부 장소 API 연동과 데이터 신뢰도 이슈가 있으므로 V1 이후 확장으로 둔다.
- 최근 방문 여부와 정교한 개인화 랭킹은 방문 기록 데이터가 충분히 쌓인 뒤 확장한다.
- 가격대, 웨이팅, 예약 필요 여부는 우선 태그로 표현하고, 구조화 필터는 이후 검토한다.
- 공개 리뷰 점수 기반 랭킹을 Pin Plate의 핵심 차별화로 삼지 않는다.

확인해야 할 주요 시나리오:

- 현재 위치 권한이 있을 때 기본 반경 `1km` 안의 주변 음식점/카페 후보를 볼 수 있다.
- 신규 사용자처럼 저장 장소가 없어도 추천 화면은 빈 화면이 아니라 주변 후보를 보여준다.
- 추천 결과 중 이미 저장한 장소는 내 상태와 저장 여부가 표시된다.
- 추천 결과 중 저장하지 않은 장소는 바로 `가보고싶음`으로 저장할 수 있다.
- `저장 안 한 곳` 칩은 이미 저장한 장소를 제외한 주변 후보를 보여준다.
- `다시 갈 만한 곳` 칩은 저장 장소가 있을 때만 `다시가고싶음` 또는 높은 별점 장소를 강조한다.
- 영업 여부나 최근 방문 여부가 없어도 V1 추천 기능은 동작한다.

### 공유 정책

사용자가 공유 버튼을 누르면 어떤 기준으로 공유 지도를 만들지 직접 선택할 수 있게 한다.

- 특정 태그 기반: 데이트, 작업하기 좋은 카페, 부모님과 가기 좋은 곳
- 특정 지역 기반: 성수 추천, 망원 맛집, 제주 저장 장소
- 특정 상태 기반: 추천하고 싶은 곳, 다시 가고 싶은 곳
- 선택한 장소 기반: 사용자가 체크한 장소만 모은 맞춤 리스트

공유 지도는 공개 링크로 접근할 수 있게 한다. 링크를 받은 사람은 로그인하지 않아도 공유된 장소를 지도 중심으로 먼저 보고, 장소 목록은 저장과 상세 확인을 돕는 보조 패널로 사용할 수 있어야 한다.

받는 사람이 `내 지도에 저장`을 누르면 가입 화면으로 먼저 막지 않고 즉시 저장 상태를 표시한다. 단, 비로그인 저장은 서버 계정에 귀속되는 영구 저장이 아니라 현재 브라우저/기기 기준의 임시 저장으로 정의한다.

공유 지도의 수신자 흐름은 다음과 같다.

1. 공유 링크를 열어 지도와 장소 목록을 본다.
2. 마음에 드는 장소를 `내 지도에 저장`한다.
3. 저장된 장소는 기본적으로 `가보고싶음` 상태가 된다.
4. 실제 방문 후 `다녀옴` 또는 `다시가고싶음`으로 바꾸고 별점, 메모, 사진을 남긴다.
5. 자기 장소가 쌓이면 다시 공유 지도를 만든다.

공유 지도 저장 정책:

- 지도 열람과 임시 저장은 가입 없이 가능하다.
- 공유 지도에서 저장한 장소의 기본 상태는 `가보고싶음`으로 둔다.
- 공유자의 상태, 별점, 평균 별점은 추천 맥락으로 보여줄 수 있지만, 받는 사람의 별점이나 방문 기록으로 그대로 복사하지 않는다.
- 영구 저장, 기기 간 동기화, 앱 재설치나 브라우저 데이터 삭제 이후 보존은 가입/로그인 이후에만 가능하다.
- 가입 유도는 저장을 막는 장벽이 아니라, 이미 저장한 장소를 유지하기 위한 보존 제안으로 표현한다.
- V1에서는 익명 사용자 데이터를 서버에 장기 저장하지 않고, 게스트 저장은 로컬 저장 기반으로 둔다.
- V1 공유 지도는 공개 링크 기반으로 만들고, 비밀번호, 만료일, 받는 사람별 권한 관리는 제외한다.

확인해야 할 주요 시나리오:

- 로그아웃 상태로 공유 링크에 접속해 지도와 장소 목록을 볼 수 있다.
- 로그아웃 상태에서 `내 지도에 저장`을 누르면 임시 저장 상태가 표시된다.
- 공유 지도의 장소를 저장하면 방문 전 후보로 저장되고, 공유자의 별점이 내 평균 별점에 반영되지 않는다.
- 임시 저장 후 가입 CTA가 표시되지만, 지도 탐색은 계속 가능하다.
- 가입/로그인하면 임시 저장한 장소를 계정 저장 장소로 가져올 수 있다.
- 다른 기기나 브라우저에서는 비로그인 임시 저장 데이터가 자동 동기화되지 않는다.

### 리포트 정책

취향 리포트는 사용자의 기록을 요약하되, 상세 DB 스키마, API, 컴포넌트 구현 계획은 이 문서에 두지 않는다. 구체 구현은 후속 구현 계획 문서로 분리한다.

제품 정책 수준에서 리포트는 다음 원칙을 따른다.

- V1은 AI 없이 `상태 + 태그 + 별점 + 장소 데이터`를 집계해 보여주는 결정적 리포트로 만든다.
- 같은 입력에는 항상 같은 리포트 결과가 나와야 한다.
- `가보고싶음`은 방문 전 후보이므로 평균 별점 계산에서 제외한다.
- 평균 별점, 많이 쓴 태그, 별점 높은 태그, 많이 저장한 지역, 최근 활동, 다음 행동 카드를 제공한다.
- 다음 행동 카드는 오래된 `가보고싶음`, 별점이 높은 재방문 후보, 사진 또는 메모가 비어 있는 방문 기록처럼 사용자가 바로 행동할 수 있는 항목을 우선한다.
- 마이페이지 메인에는 저장 장소 수, 평균 별점, 자주 쓰는 태그처럼 가벼운 미리보기만 보여주고 `취향 리포트 보기`로 연결한다.
- `/my-page/report`는 취향 스냅샷, 태그/지역/별점 통계, 다음 행동 카드를 모두 보여주는 본 화면으로 둔다.
- 지도와 리스트에는 전체 리포트를 직접 노출하지 않고 `다시 가고 싶은 곳`, `자주 저장한 지역`, `사진/메모가 비어 있는 기록`처럼 행동에 필요한 일부 요약만 활용한다.

후속 구현 계획 문서에서 별도로 다룰 항목:

- DB 필드와 마이그레이션 SQL
- `Post`, create/update payload 등 공개 타입
- 리포트 집계 함수의 입출력 타입과 정렬 규칙
- 게시글 작성/수정 UX의 상태별 검증 규칙
- 리포트 화면 컴포넌트 구조
- 단위 테스트, 화면 테스트, 수동 확인 항목

## 7. 기술/문서 정리 필요 사항

### 지도 provider 설명 불일치

README는 주요 지도 서비스가 Naver Maps API V3라고 설명한다. 반면 실제 웹 코드의 지도 컴포넌트는 `@vis.gl/react-google-maps`와 Google Maps 기반으로 구성된 부분이 확인된다.

정리 방향 중 하나를 선택해야 한다.

- 실제 구현이 Google Maps라면 README와 기술 스택 설명을 Google Maps 기준으로 업데이트한다.
- 제품/국내 지도 전략상 Naver Maps를 유지하려는 의도라면 실제 코드와 환경 변수를 Naver Maps 기준으로 정리한다.
- 장소 검색은 Kakao Local API를 계속 사용할 수 있으므로, 지도 provider와 장소 검색 provider를 문서에서 분리해 설명한다.

이 부분은 기능 개발보다 먼저 정리해 두는 편이 이후 기획 문서와 구현 논의의 혼선을 줄인다.

## 8. 비목표

- 특정 기능의 DB 스키마, API, 컴포넌트 상세 구현 계획은 이 문서 범위에 포함하지 않는다.
- 유사 서비스 전체 벤치마크를 길게 정리하지 않는다.
- 공개 리뷰/소셜 피드 중심 서비스로의 전환은 현재 방향과 다르므로 우선순위에서 제외한다.
