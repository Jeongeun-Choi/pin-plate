# Claude Code 스킬: tiki-taka와 opus-sonnet으로 AI 개발 워크플로우 개선하기

## 들어가며: AI가 코드를 작성할 때의 문제점

Claude와 같은 AI가 코드 작업을 할 때, 흔히 다음과 같은 문제가 발생한다:

1. **설계 검토 부재** — AI가 설계 없이 바로 구현을 시작하면, 아키텍처 문제나 엣지 케이스를 놓치기 쉽다
2. **리팩터링 비용** — 설계 결함은 구현 중간에 발견될수록 수정 비용이 기하급수적으로 증가한다
3. **단일 관점** — 한 AI만 진행하면 blind spot이 남는다. 다른 관점의 리뷰가 필요하다
4. **모델 효율성** — 복잡한 설계 결정과 단순한 코드 작성 둘 다 고성능(비싼) 모델로 처리하면 비용이 높다

이런 문제를 해결하기 위해 pin-plate 프로젝트는 두 가지 커스텀 Claude Code 스킬을 개발했다:
**tiki-taka**(설계 리뷰 자동화)와 **opus-sonnet**(역할 분리 파이프라인).

---

## Tiki-taka: 설계 리뷰 자동화

### 도입 목적

복잡한 기능을 개발할 때, AI가 혼자 설계·구현하면 놓치기 쉬운 아키텍처 문제를 **설계 단계에서 사전에 발견**하려는 목적이다.

특히 다음 상황에서 효과적이다:

- 변경해야 할 파일이 **6개 이상** 인 작업
- **새로운 기능** 추가 (기존 코드 탐사 필요)
- **아키텍처 변경** (여러 계층에 영향)
- **복잡한 상태 관리** (여러 스토어·쿼리 상호작용)

---

### 동작 원리

tiki-taka는 **"설계 → 리뷰 → 구현"** 의 3단계 프로세스로 진행되며, 핵심은 구현 **전**에 설계를 검증하는 것이다.

#### Phase 1: 설계 초안 작성

작업이 요청되면, Claude가 먼저 코드를 작성하지 않고 **구현 계획서 초안을 작성**한다.

계획서에 포함되는 항목:

- **변경·생성할 파일 목록**: 이 작업으로 인해 영향을 받을 파일들 (컴포넌트, 스타일, API, 타입 정의 등)
- **각 파일의 작업 범위**: 어떤 파일에서 무엇을 할 것인가
  - 새로운 인터페이스나 타입 정의
  - 함수/메서드 시그니처
  - 상태 관리 구조 (Jotai 스토어, React Query 쿼리 등)
- **예상되는 엣지 케이스와 에러 처리**: 정상 흐름이 아닌 예외 상황들을 어떻게 처리할 것인가
- **성능·보안 고려사항**: 이 변경이 성능이나 보안에 미칠 수 있는 영향
- **pin-plate 컨벤션 준수**: Server/Client Components 분리, 훅 순서, 네이밍 규칙 등

**예시:** "사용자 팔로우 기능 추가"라면, 계획서는 다음을 포함해야 한다:
- 어느 컴포넌트가 팔로우 버튼을 렌더링할 것인가
- API 엔드포인트는 어떻게 설계할 것인가
- 팔로우 상태는 어디서 관리할 것인가 (React Query? Jotai?)
- 이미 팔로우한 사용자를 다시 팔로우하려 하면? (엣지 케이스)
- 네트워크 오류 시 어떻게 처리할 것인가

#### Phase 2: 티키타카 리뷰 루프 (최대 3라운드)

계획서가 완성되면, **Claude와 Codex가 번갈아가며 검토**하는 리뷰 루프가 시작된다. "티키타카"라는 이름도 여기서 비롯되었다 (탁구처럼 주고받으며 진행).

**Round 1에서 이런 과정이 반복된다:**

1. **Codex의 검토**: 계획서를 객관적인 관점에서 검토한다.
   - 아키텍처가 건전한가? (예: 순환 의존성이 없는가, 책임이 명확하게 분리되었는가)
   - pin-plate의 컨벤션을 따르고 있는가? (예: Server Components인가, 훅 순서가 맞는가)
   - 놓친 엣지 케이스가 있는가? (예: 동시에 여러 요청이 들어올 때는, 서버 오류는)
   - 타입 안전성이 보장되는가? (예: `any` 사용은 없는가)
   - 성능 문제는 없는가? (예: 불필요한 리렌더링은 없는가)
   
   검토를 마친 후 Codex는 결론을 내린다:
   - `LGTM` (Looks Good To Me): 이 계획으로 구현해도 괜찮다
   - `NEEDS_CHANGES`: 수정할 부분이 있다

2. **Claude의 응답**: Codex의 의견을 분석한다.
   - 타당한 지적이라면? → 계획서를 수정하고, 수정 이유를 명확히 설명
   - 필요 없거나 틀린 지적이라면? → 근거를 들어 명확하게 반박
   
   응답 후 Claude도 결론을 낸다:
   - `LGTM`: Codex의 의견을 받아들였거나, 이미 고려했다
   - `NEEDS_CHANGES`: 아직 합의에 이르지 못했다

3. **종료 조건 확인**: 
   - 양측 모두 `LGTM`이면? → 설계 확정, Phase 3으로 진행
   - 하나라도 `NEEDS_CHANGES`면? → 다음 라운드로 진행 (최대 3라운드)

**Round 2, 3**: Round 1과 동일한 과정을 반복한다. 이전 라운드의 의견도 함께 검토하므로, 논의가 점점 수렴된다.

만약 3라운드 후에도 합의에 이르지 못하면, 미결정 사항을 명시하고 사용자가 최종 판단한다.

**왜 이렇게 할까?**

- **설계 검증**: 코드를 작성하기 전에 아키텍처 결함을 찾음 (코드 작성 후 발견하면 리팩터링 비용이 크다)
- **두 관점의 크로스 체크**: Claude가 놓친 부분을 Codex가 잡고, 반대도 마찬가지 (blind spot 감소)
- **프로젝트 규칙 자동화**: CLAUDE.md의 규칙들(Server Components, 훅 순서 등)을 매번 체크하므로, 사후 코드 리뷰에서 이런 항목들을 다시 지적할 필요가 없다

#### Phase 3: 구현

양측이 모두 `LGTM`을 제시하면, 설계가 **최종 확정**된다. 이제 Claude가 합의된 설계를 기반으로 **실제 코드를 구현**한다.

계획서가 이미 작성되어 있으므로, 구현은 "무엇을 만들지" 고민 없이 "어떻게 만들지"에만 집중할 수 있다. 결과적으로:
- 구현 속도가 빨라진다
- 설계와 구현의 괴리가 줄어든다 (계획서대로 만들기만 하면 됨)

---

### 적용 기준

tiki-taka를 **자동으로 적용하지 않는** 경우:

- 변경 파일 **5개 이하** 로 완결되는 작업
- **버그 수정** (원인 파악 완료, 범위 한정)
- **단순 스타일 변경** (CSS, 디자인 토큰, Vanilla Extract)
- **문서 수정** (README, 주석, 마크다운)
- **명백히 단순한 작업** (변수명 변경, 한 줄 수정 등)

이외의 복잡하거나 새로운 기능 구현, 아키텍처 변경은 **반드시** tiki-taka를 실행한다.

---

### 이점

#### 1. 설계 결함을 사전에 발견
코드를 작성하기 전에 문제를 찾으므로, 리팩터링 비용이 대폭 줄어든다.

#### 2. 두 AI의 크로스 리뷰
Claude와 Codex라는 서로 다른 관점이 만나면서, 한 쪽이 놓칠 수 있는 blind spot을 줄인다.

#### 3. 프로젝트 컨벤션 준수 자동화
CLAUDE.md의 규칙 (Server Components, 훅 순서, 네이밍, 접근성 등)을 체크하므로,
사후 코드 리뷰에서 컨벤션 위반을 발견하는 낭비를 줄인다.

#### 4. 구현 품질 향상
설계가 정해진 상태에서 구현하면, "뭘 만들지" 고민 없이 "어떻게 만들지"에 집중할 수 있다.

## Opus-Sonnet: 역할 분리 파이프라인

### 도입 목적

설계와 구현은 요구하는 능력과 효율성이 다르다:

- **설계**: 아키텍처 결정, trade-off 분석, 프로젝트 규칙 이해 → **고비용, 고품질 모델 필요**
- **구현**: 코드 작성, 반복적 수정, 포매팅 → **빠르고 저렴한 모델으로 충분**

**opus-sonnet**은 이를 모델 수준에서 분리하여 **설계 품질은 유지하면서 비용과 속도를 개선**하는 것을 목표로 한다.

---

### 동작 원리

opus-sonnet은 **설계·리뷰·구현·검증**을 모델 수준에서 분리하는 4단계 파이프라인이다.

#### Phase 1-2: Opus의 설계 + tiki-taka 리뷰

**Opus (claude-opus-4-6)** 가 주도한다. 처음 두 단계는 tiki-taka와 동일하지만, **자동화**된다는 점이 다르다.

1. **요청 분석 및 설계 초안**
   
   사용자의 요청이 들어오면, Opus가 먼저 코드를 작성하지 않고 구현 계획서를 작성한다. 이 과정은 tiki-taka의 Phase 1과 동일하다:
   - 변경할 파일 목록 파악
   - 각 파일의 구현 범위 정의
   - 엣지 케이스, 성능, 보안 고려
   - 프로젝트 컨벤션 준수 계획

2. **tiki-taka 자동 호출**
   
   계획서가 작성되면, 이를 기반으로 **tiki-taka 스킬을 자동으로 호출**한다. 사용자의 개입 없이 내부 서브루틴으로 실행되는 것이다.
   
   - Codex가 계획서 검토
   - Opus가 의견 분석 및 대응
   - 최대 3라운드 진행
   - 양측 LGTM 달성 시 설계 확정

**결과:** 최종 확정된 설계가 계획 파일(`plan.md`)로 저장된다. 이 시점에서 설계는 **Opus + Codex의 합의**를 받은 상태다.

---

#### Phase 3: 작업 분해 및 병렬 구현

설계가 확정되면, 이제 Opus는 **실장 책임자** 역할로 전환된다.

**3-1. 작업 분해**

Opus가 계획서를 읽고, 이를 **독립적으로 실행 가능한 작업 단위**로 분해한다.

예시: "사용자 프로필 페이지 개선" 계획서가 다음과 같다면:

```
- UserProfile 컴포넌트 업데이트
- FollowerTab 컴포넌트 구현
- FollowingTab 컴포넌트 구현
- SettingsModal 컴포넌트 구현
- useUserProfile 훅 구현 (API 쿼리)
- useFollowUser 훅 구현 (팔로우 뮤테이션)
- notificationStore 구현 (Jotai)
```

Opus는 이를 다음과 같이 분해한다:

- **Task A (독립)**: FollowerTab, FollowingTab 컴포넌트 구현 → Sonnet-1
- **Task B (독립)**: SettingsModal 컴포넌트 구현 → Sonnet-2
- **Task C (의존 없음)**: notificationStore (Jotai) 구현 → Sonnet-3
- **Task D (Task C 이후)**: useFollowUser 훅 구현 (store 필요) → Sonnet-4
- **Task E (Task A/D 이후)**: UserProfile 컴포넌트 업데이트 → Sonnet-1 or Opus

**분해 기준:**
- **독립적인 작업** (다른 작업에 의존하지 않음) → 병렬 실행
- **의존성이 있는 작업** (특정 파일/인터페이스가 먼저 생성되어야 함) → 순차 실행
- **작업이 너무 적거나 강하게 결합** (3개 미만 or 매우 복잡) → 한 Sonnet에 통합

**3-2. Sonnet 에이전트 디스패치**

각 Task마다 **Sonnet 에이전트(claude-sonnet-4-6)** 를 디스패치한다:

- **병렬 디스패치**: Task A, B, C는 의존성이 없으므로 동시에 3개의 Sonnet을 실행
- **순차 디스패치**: Task D는 Task C 완료 후에 시작, Task E는 Task D 완료 후 시작

각 Sonnet에게 주어지는 지시는 다음을 포함한다:

```
Task: [작업 설명]

설계 원본 (plan.md 중 해당 부분):
[구현해야 할 범위]

제약사항:
- 할당된 파일만 수정 (다른 파일은 건드리지 않음)
- CLAUDE.md 규칙 준수 (Server/Client, 훅 순서, 네이밍 등)
- TypeScript 타입 안전성 (any 금지)
- 완료 후 자신의 코드에 대해 lint, 타입체크 실행

완료 후:
- 생성/수정한 파일 목록
- 우려사항이나 불명확한 점 보고
```

**Sonnet의 특징:**
- 빠르고 저렴 (Opus 대비)
- 실제 코드 작성 능력은 충분함
- 단, 아키텍처 결정이나 복잡한 설계는 Opus가 담당
- 따라서 이미 정해진 설계를 따르기만 하면 됨

**병렬성의 강점:**

- Task A (10분) + Task B (10분) + Task C (10분)이 있다면
  - 순차 실행: 30분
  - 병렬 실행: 10분 (3배 빠름)
  
- 실제 프로젝트에서는 여러 Task가 동시에 진행되므로, 대규모 작업도 짧은 시간에 완료 가능

---

#### Phase 4: Opus의 최종 검증

모든 Sonnet 에이전트가 작업을 완료하면, **Opus가 최종 검증 및 조정**을 담당한다. 이 단계가 중요한 이유는 여러 Sonnet이 동시에 작업했을 때 발생할 수 있는 불일치를 해소하기 위함이다.

**4-1. 자동 검증**

먼저 기계적인 검증을 실행한다:

```bash
pnpm lint:fix      # ESLint + Prettier 적용
pnpm tsc --noEmit  # TypeScript 타입체크
```

이 과정에서 다음이 확인된다:
- ESLint 규칙 준수 (컨벤션, 네이밍, 사용되지 않은 변수 등)
- TypeScript 타입 안전성 (any 사용, 타입 불일치 등)
- Prettier 포매팅 일관성

**4-2. 코드 리뷰**

자동 검증 후, Opus가 **사람처럼** 코드를 읽으며 검토한다:

- **계획서 대비 완성도**: 계획서에서 약속한 기능이 모두 구현되었는가?
- **CLAUDE.md 컨벤션 준수**: 훅 순서, Server/Client 분리, 네이밍 규칙 등이 지켜졌는가?
- **엣지 케이스 처리**: 예상했던 엣지 케이스들이 제대로 처리되었는가?
- **타입 안전성**: TypeScript 타입이 명확하고 안전한가? (`unknown` 사용 여부, 타입 좁히기 등)
- **여러 Sonnet 간 일관성**: Sonnet-1이 만든 코드와 Sonnet-2가 만든 코드 스타일이 다르지는 않은가?

**4-3. 수정**

검토 결과에 따라 수정을 진행한다:

- **경미한 오류** (린트 경고, 네이밍, 간단한 타입 수정, 주석 추가 등)
  → **Opus가 직접 수정** (빠름)
  
- **대규모 수정** (로직 변경, 구조 재작성, 새 함수 추가 등)
  → **해당 Sonnet을 재호출** (최대 2회)
  
  예: Sonnet-1이 만든 컴포넌트의 상태 관리 로직이 틀렸다면,
  더 구체적인 지시와 함께 Sonnet-1을 다시 실행하게 함

**왜 이렇게 할까?**

- Opus(설계) → Sonnet(구현) → Opus(검증)의 분리로, 각 모델이 가장 잘하는 일에 집중
- Opus의 고비용은 핵심 결정에만 사용, Sonnet의 저비용은 반복 작업에 사용
- 검증 단계에서 여러 Sonnet이 만든 코드를 통합하고 검수하므로 품질 보장

---

### 이점

#### 1. 설계 품질 + 구현 속도
- Opus의 고급 설계 능력과 tiki-taka의 리뷰로 견고한 설계 보장
- Sonnet의 빠른 구현으로 시간 단축

#### 2. 병렬 처리로 인한 속도 향상
- 독립적인 작업을 여러 Sonnet이 동시에 처리
- 순차 처리 대비 2배 이상 빠를 수 있음

#### 3. 비용 절감
- 고비용 Opus는 설계·리뷰·검증에만 사용
- 저비용 Sonnet은 구현·코딩에 집중
- 평균적으로 전체 비용 30~40% 절감 가능

#### 4. 자동화된 리팩터링
- 여러 Sonnet이 동시에 작업하면서 발생할 수 있는 스타일·타입 불일치 → Opus의 최종 검증으로 해소
- 사람이 수동 리뷰할 필요 없음

## 실제 적용 예시

### 시나리오: 사용자 프로필 페이지 개선

**요청**: 사용자 프로필에 팔로우/팔로워 탭, 설정 모달, 알림 설정 추가

**예상 영향 범위**:
- 10개 파일 수정 (컴포넌트, 스타일, 상태관리, API)
- 새 훅 2개
- 데이터베이스 스키마 변경 불가 (기존 필드 활용)

**선택:**
→ **opus-sonnet 사용**

**이유:**
- 변경 파일 10개 (6개 초과) → tiki-taka 후보 ✓
- 여러 독립 작업 (탭 UI, 모달, 알림 설정)으로 분리 가능 → 병렬화 가능 ✓
- 팀이 빠른 완료를 원함 → opus-sonnet의 속도 메리트 ✓

**진행:**

1. **Opus**: 요청 분석 → 설계 초안 (각 탭·모달·설정의 컴포넌트 구조, 상태 관리 전략)
2. **tiki-taka**: 자동 호출 → Codex 리뷰 → 설계 확정
3. **작업 분해**: 
   - Task A: 탭 UI 컴포넌트 (Sonnet-A)
   - Task B: 팔로우/팔로워 로직 (Sonnet-B)
   - Task C: 설정 모달 + API (Sonnet-C)
   - Task D: 상태 관리 및 쿼리 (Sonnet-D, Task B/C 이후)
4. **병렬 실행**: Sonnet-A, B, C 동시 진행, Task D는 대기
5. **Opus 검증**: 전체 코드 lint/타입체크, 컨벤션 확인

**결과:**
- 설계 리뷰로 아키텍처 결함 사전 방지
- 병렬 처리로 약 40% 시간 단축
- Opus 설계 + Sonnet 속도 결합

## 마치며: AI 워크플로우 설계의 미래

tiki-taka와 opus-sonnet은 단순한 스킬이 아니라, **AI-assisted 개발의 본질을 담은 워크플로우**다:

1. **단계 분리** (설계 → 리뷰 → 구현 → 검증)로 각 단계에서 실수를 최소화
2. **역할 분리** (고비용 모델은 결정, 저비용 모델은 실행)로 비용·품질 최적화
3. **자동화된 리뷰** (Codex의 객관적 관점)로 blind spot 제거
4. **병렬 처리**로 대규모 작업의 속도 향상

이는 프로젝트의 규모와 특성에 따라 언제든 커스터마이징할 수 있다. 

예를 들어:
- 설계 리뷰를 더 엄격하게 하려면? → tiki-taka 라운드 제한 제거
- 구현 속도를 극대화하려면? → opus-sonnet의 병렬 작업 수 증가
- 특정 도메인에만 적용하려면? → 스킬 적용 조건 커스터마이징

**결론**: AI 개발 도구는 "얼마나 똑똑한가"가 아니라, **"얼마나 체계적으로 오류를 줄이고 효율을 높이는가"** 가 핵심이다. tiki-taka와 opus-sonnet은 pin-plate 팀이 만들어낸 그러한 체계의 좋은 사례다.
