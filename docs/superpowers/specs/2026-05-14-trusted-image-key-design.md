# Trusted Image Key Design

## Goal

Pin Plate image references should be trusted because they came from the upload server, not because a client sent a URL-shaped string. This protects logged-in posts, guest posts, shared maps, and Open Graph metadata from arbitrary external image URLs and blind SSRF.

## Current Problem

The upload API creates S3 object keys and returns public URLs, but post creation and shared map creation still pass image URL strings through client-controlled payloads. A malicious client can bypass the UI and send an arbitrary `image_urls`, `first_image`, or `cover_image_url` value. The recent Open Graph fallback check then server-fetches `cover_image_url`, so any user-controlled external URL can become a server-side request.

## Product Constraints

- Guest writing and image upload must keep working.
- Guest posts remain local-first in `localStorage` until the user signs in.
- Existing `image_urls` data must keep rendering.
- This change should not require users to log in before uploading.
- S3 and CloudFront do not decide what can be stored in the database; app/server validation does.

## Design

Use the S3 object key as the trusted internal image reference.

`/api/image` will return both:

- `imageKey`: the S3 object key generated by the server, such as `uploads/users/{userId}/{uuid}.webp` or `uploads/guests/{guestId}/{uuid}.webp`
- `publicUrl`: the display URL derived from the configured image public base URL and `imageKey`

Client UI can keep displaying `publicUrl`, but persistence paths should carry `imageKeys` when saving posts or syncing guest posts. A trusted server boundary must reconstruct public URLs from validated keys instead of trusting arbitrary submitted URLs. Client-only validation is not security because a malicious browser can call Supabase directly.

## Trusted Key Rules

An image key is valid only when all of these are true:

- It is a relative S3 object key, not a URL.
- It starts with `uploads/users/{userId}/` for logged-in users.
- It starts with `uploads/guests/{guestId}/` for guest users.
- It ends with an allowed image extension: `.jpg`, `.png`, `.webp`, or `.gif`.
- It does not contain path traversal such as `..`.

## Public URL Rules

The canonical public URL is built from:

```txt
IMAGE_PUBLIC_BASE_URL + "/" + imageKey
```

For production, the preferred value is the configured CloudFront custom domain.
Examples should use placeholder domains rather than real upload origins:

```txt
IMAGE_PUBLIC_BASE_URL=https://image.example.test
```

Before CloudFront is ready, the S3 public origin can be used:

```txt
IMAGE_PUBLIC_BASE_URL=https://<bucket>.s3.<region>.amazonaws.com
```

Open Graph metadata may still receive old `cover_image_url` data, so it should accept only URLs whose origin matches the configured public image base and whose path maps to a valid upload key. Everything else falls back to `/og-default.png` without a network request.

## Data Model Compatibility

Existing UI and database columns can keep using `image_urls` during the transition, but server-backed writes need one trusted enforcement point. The preferred implementation is a Next.js API boundary or database RPC that receives `image_keys`, validates actor ownership, derives `image_urls`, and writes only sanitized image data. If a real `image_keys` column is added later, the same boundary can store keys too. Direct client writes that can set arbitrary `image_urls` must be removed from app code and eventually blocked by RLS/RPC policy.

Recommended migration path:

1. Add image reference utilities.
2. Return `imageKey` from `/api/image`.
3. Track uploaded photos as `{ key, url }` in post forms.
4. Store guest posts with both `image_urls` and `image_keys`.
5. Route server-backed post create/update through a trusted API/RPC that derives `image_urls` from validated `image_keys`.
6. Sync guest posts using `image_keys` when available; the server route can read the HttpOnly guest upload cookie and accept matching `uploads/guests/{guestId}/` keys during account sync.
7. Validate shared map images before metadata fetch.
8. Add DB/RLS follow-up that blocks direct arbitrary `image_urls` writes outside the trusted path.

## Shared Map Policy

Shared map creation should not trust arbitrary `first_image` or `cover_image_url` values from client payloads. The safest final model is for the database RPC to receive source place IDs, verify ownership with `auth.uid()`, and compute the snapshot image fields from the user's stored posts/places.

Until that RPC is changed, the app should at least validate cover and place image URLs against the trusted public image policy before using them in Open Graph metadata. Invalid images should silently fall back to `/og-default.png`.

## Error Handling

- Invalid image keys are rejected before saving server-backed posts.
- Invalid guest image keys are ignored during local parsing/sync rather than breaking guest post loading.
- Invalid shared map cover URLs never trigger `fetch`; they use the default OG image.
- Existing old `image_urls` render normally, but are not treated as trusted input for new server-side fetches.

## Test Strategy

- Unit-test key validation and URL generation.
- Test `/api/image` response includes `imageKey` and `publicUrl`.
- Test post upload state keeps key and URL together.
- Test guest post parsing supports missing legacy `image_keys`.
- Test shared metadata does not fetch non-allowlisted cover URLs.
- Test allowlisted cover URLs still use `HEAD` and fall back on failure.

## Non-Goals

- Do not require login for guest upload.
- Do not add malware scanning in this step.
- Do not rewrite all existing database rows.
- Do not make S3 bucket policy the only enforcement layer.
